Visual Style

Usability

Adhere to standards and be consistent

Adhering to standards for interface elements and interactions helps our users to be able to intuitively use your interface. This is because they can build upon existing knowledge.

Even experienced and tech-savvy users benefit from standards: They, too, learn and adopt it better and they feel more save when doing critical actions.

Show how one can interact with an widget

Show the users what they can do and what they can archive by that. Don't leave users wondering what a function does and how it can be activated.

Using standard elements is essential in communicating how an element can be used:

EXAMPLE: Buttons, Links afford clicks; sliders afford clicks on slider OR dragging. "CRTL+A" in a menu entry affords pressing the keys.

To leave out such signifiers may be tempting due to saving space, adding functions without changing the UI or keeping it visually clean. However, it means that you leave your users guessing what the UI does.

Use plain, easy to get lables or icons to show what the use of an element does.

…Example… OK -- Save -- Publish now.

Also with element actions, there are some standards. EXAMPLE: Link leads to new page. ... button opens a new dialog.

Prevent Errors and alliviate them

In many cases you can prevent problems occuring in the first place. If a user could make invalid input, try to provide suggestions so the user can choose from a list of valid choices.

If it does not make sense to execute an action, you can deactive these functions. Most components like buttons provide a deactivated state which can be used to show that using this does not make sense at the moment.

example from Visual Editor: If you did not change text, your publish button is gray

Make sure that the user can understand why the button is disabled. This is usually done by being close to the offending input. If it is not, you can consider adding a tooltip or even a help text.

If an error occurs, inform the user in a way that allows the user to understand and fix the error. Avoid jargon and technical infos and focus on what the user can do now.

Care for keyboard usage

Make the functionality accessible via keyboard, so the interface is usable without a mouse. This benefits two user groups:

  • Users with motor or visual disabilities who can't use a mouse
  • Users who prefer keyboard use, since it is faster
